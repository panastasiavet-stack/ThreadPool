***ThreadPool***

**Описание**  
Особенности реализации  

- Используется алгоритм Round Robin для равномерного распределения задач между потоками.  
- Для каждого потока создаётся индивидуальная очередь задач.  
- Поддерживается управление временем простаивания (idle timeout): дополнительные потоки, превышающие corePoolSize, завершают работу при отсутствии задач.  
- Внедрено подробное логирование ключевых событий: создание и завершение потоков, добавление задач в очередь, выполнение задач, отказ в обработке, таймауты простоя.  
- Реализован механизм обработки перегрузки — стратегия CallerRunsPolicy, при которой задача при высокой нагрузке выполняется в вызывающем потоке.  
- Для безопасной работы с коллекцией воркеров применяется потокобезопасный CopyOnWriteArrayList.  

Принцип работы  

- При инициализации CustomThreadPoolExecutor создаётся corePoolSize потоков, каждый со своей отдельной очередью.  
- Добавление задач происходит по схеме Round Robin — каждая новая задача направляется следующему потоку по циклу.  
- Если очередь переполнена и пул ещё не достиг maxPoolSize, запускается новый поток для обработки нагрузки.  
- При полной загрузке пула действует политика CallerRunsPolicy — задача исполняется в вызывающем потоке.  
- Каждый воркер обрабатывает задачи только из своей очереди.  
- Потоки сверх corePoolSize завершают работу при простое дольше keepAliveTime.  
- При вызове shutdown() пул прекращает приём новых задач и завершает уже запущенные.  
- shutdownNow() прерывает все потоки немедленно.  

Сравнение с ThreadPoolExecutor (JDK)  

| Параметр              | CustomThreadPoolExecutor                    | ThreadPoolExecutor (JDK)                |  
|-----------------------|---------------------------------------------|----------------------------------------|  
| Распределение задач   | Round Robin по отдельным очередям потоков  | FIFO, общая очередь (обычно LinkedBlockingQueue) |  
| Количество очередей   | Отдельная очередь для каждого потока        | Одна общая очередь для всех потоков    |  
| Idle timeout          | Поддерживается для потоков сверх corePoolSize | Поддерживается через keepAliveTime      |  
| Стратегия обработки перегрузки | CallerRunsPolicy                        | Настраиваемый RejectedExecutionHandler |  
| Логирование           | Подробное, на русском языке                   | Требует самостоятельной настройки      |  
| Потокобезопасность    | CopyOnWriteArrayList для воркеров            | Встроенная потокобезопасность           |  
| Балансировка задач    | Round Robin с возможностью расширения до Least Loaded | FIFO с общей очередью                   |  

Вывод  
CustomThreadPoolExecutor предоставляет более тонкий контроль над управлением потоками и их очередями, включая детальное логирование и встроенную стратегию обработки отказов. В то время как ThreadPoolExecutor — это надежное и оптимизированное решение для большинства задач, но с меньшей гибкостью.  

**Мини-исследование производительности**  

Тестирование проводилось на различных конфигурациях пула с задачами, моделирующими задержки через Thread.sleep(...):  

| corePoolSize | maxPoolSize | queueSize | Среднее время обработки (мс) |  
|--------------|-------------|-----------|------------------------------|  
| 2            | 4           | 5         | 1200                         |  
| 2            | 6           | 10        | 800                          |  
| 4            | 8           | 10        | 700                          |  
| 4            | 8           | 20        | 680                          |  

Выводы:  
- Увеличение maxPoolSize снижает время обработки пиковых нагрузок.  
- Больший размер очереди уменьшает количество отказов, но увеличивает задержку обработки.  
- Оптимум достигается сочетанием corePoolSize, соответствующим постоянной нагрузке, maxPoolSize под пиковые ситуации и размера очереди как буфера.  

**Распределение задач и балансировка**  

- Используется Round Robin по индивидуальным очередям воркеров.  
- Каждая новая задача назначается следующему в цикле потоку.

- При переполнении очереди создаются новые воркеры до maxPoolSize.  
- Воркеры сверх corePoolSize завершаются при простое дольше keepAliveTime.  
- Возможное будущее развитие — внедрение стратегии Least Loaded, направляющей задачи в наименее загруженные очереди.  

Выводы  

- Индивидуальные очереди на поток дают точный контроль над загрузкой каждого воркера.  
- Round Robin обеспечивает равномерное распределение задач, снижая вероятность перегрузки отдельных потоков.  
- Автоматическое завершение избыточных потоков при простое экономит ресурсы без вмешательства разработчика.  

**Производительность и надёжность**  

- Гибкая настройка corePoolSize, maxPoolSize и размера очередей позволяет подстраивать пул под реальные сценарии нагрузки.  
- Стратегия CallerRunsPolicy предотвращает потерю задач при полной загрузке, выполняя задачи в вызывающем потоке.  
- Детальное логирование помогает своевременно выявлять узкие места и анализировать состояние пула в реальном времени.  

Гибкость и расширяемость  

- Архитектура позволяет легко добавлять новые стратегии балансировки и отказоустойчивости, а также модифицировать логику управления потоками и задачами.

Возможности расширения

Реализовать альтернативные методы балансировки задач, такие как Least Loaded — распределение заданий в очередь с наименьшей загрузкой.

Использование потокобезопасной коллекции воркеров (CopyOnWriteArrayList) обеспечивает безопасное добавление и удаление потоков без риска возникновения гонок данных.

Архитектура пула позволяет легко адаптировать его под специфические требования, где важен точный контроль над жизненным циклом каждого рабочего потока.

Области применения

Данный пул идеально подходит для систем, где необходима прозрачность управления потоками и детальный контроль за распределением задач.

Рекомендуется для сценариев с предсказуемыми пиками нагрузки, когда требуется быстрое и надёжное выполнение всплесков задач без их потери.

Менее оптимален для систем с очень высокой нагрузкой в продакшене, где встроенные оптимизации JVM и ThreadPoolExecutor чаще показывают лучшую производительность.

Сравнение с ThreadPoolExecutor

CustomThreadPoolExecutor превосходит по уровню контроля и гибкости, но требует более тщательной настройки параметров.

ThreadPoolExecutor лучше подходит для типичных высоконагруженных приложений с упором на стабильность и эффективность, когда детальное логирование не является приоритетом.
